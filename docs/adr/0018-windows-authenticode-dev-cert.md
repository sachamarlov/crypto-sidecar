# 0018 — Windows Authenticode: self-signed dev cert for academic delivery

- Status: accepted
- Date: 2026-04-27
- Deciders: @sachamarlov, Claude Opus 4.7 (1M context)
- Tags: [packaging, distribution, security, signing, ci]
- Related: ADR-0001 (Tauri 2 + Python sidecar), ADR-0012 (PyInstaller
  for MVP), ADR-0016 (Tauri sidecar IPC security).
- Tracks: Phase I (Build & distribution finale).

## Context

The GuardiaBox `.exe` produced by `pnpm tauri build` for Windows
needs a code-signing strategy by 2026-04-29 (academic delivery).
Three signing tiers exist on the Authenticode market:

1. **EV (Extended Validation) cert** — issued by a CA after a
   physical-evidence vetting process, USB-token-bound private key.
   Bypasses Microsoft SmartScreen reputation accrual entirely.
   ~600 €/year, 1–3 weeks lead time.
2. **OV (Organisation Validated) cert** — issued by a CA after a
   business-registration check, software-bound private key. Earns
   SmartScreen reputation gradually (typically a few weeks of
   user installs). ~250 €/year, ~3–5 days lead time.
3. **Self-signed dev cert** — generated locally with `New-SelfSigned
Certificate` or `signtool`. Free, immediate. **Always triggers a
   SmartScreen "unknown publisher" warning** because the cert chain
   does not anchor in the Microsoft Trusted Root Program.

The academic deliverable is a one-shot demo on a known machine
(reviewer + jury). It is not a public release.

## Considered options

### A. Ship unsigned (no Authenticode at all)

**Pros**

- Zero CI complexity. The `.exe` produced by `tauri build` is the
  artefact users get.
- No certificate management, no expiry tracking.

**Cons**

- Windows Defender SmartScreen displays "Windows protected your PC
  / Unrecognized app" with a tiny "More info" → "Run anyway" path
  that most users (including academic reviewers) do not find on
  first try.
- Looks unprofessional for a project whose core promise is
  cryptographic security. A reviewer who has to click through an
  AV warning to launch a vault app receives a contradictory signal.

### B. Self-signed dev cert (**chosen**)

**Pros**

- Free and reproducible. The cert + chain can be regenerated by any
  developer who clones the repo, on any Windows host with PowerShell
  ≥ 5.1.
- The signed `.exe` carries an Authenticode signature block that
  `signtool verify /pa /v` validates against the local cert store.
  This proves the binary has not been tampered with **after the
  developer signed it** — that is the security property we want.
- Demonstrates DevSecOps discipline (CI step, secret management,
  signing pipeline) without the academic project paying €250+ for
  a one-day demo.
- Lets the demo machine import the signing cert into its
  Trusted Publishers store before launch, eliminating the
  SmartScreen warning for that one machine. A documented
  pre-flight step, not a hack.

**Cons**

- On any other Windows machine, the SmartScreen warning still
  appears. We must document this honestly in the README so a
  hypothetical user post-academic does not feel deceived.
- Cert renewal: self-signed certs expire (default 1 year). Future
  releases will need a re-issue, tracked in the Phase I follow-up
  list.
- The .exe will not gain SmartScreen reputation — every fresh
  install on a new machine triggers the same warning.

### C. OV cert via SSL.com / Sectigo / DigiCert

**Pros**

- Gradual SmartScreen reputation build-up — after ~5 000 installs
  the warning disappears for everyone.
- Real third-party validation of the publisher identity.

**Cons**

- ~250 €/year. The academic project budget is zero.
- 3–5 day lead time we cannot afford this close to delivery.
- Solo-developer registration friction: SSL.com, Sectigo etc.
  require a registered business or a notarised individual identity
  proof. Out of scope for an academic GCS2 project.

### D. EV cert with USB token

**Pros**

- Strongest possible Authenticode trust tier.
- Skips SmartScreen reputation gating entirely from day one.

**Cons**

- ~600 €/year. Out of scope.
- USB token cannot live in a CI runner — would require a self-hosted
  runner with the token plugged in, doubling the infra surface.
- 1–3 week lead time. Out of scope.

## Decision

Adopt **Option B — Self-signed dev cert**.

### Concrete invariants

1. **Cert generation** documented in
   `docs/security/AUTHENTICODE_DEV_CERT.md`:

   ```powershell
   $cert = New-SelfSignedCertificate `
     -Type CodeSigningCert `
     -Subject "CN=GuardiaBox Dev (Sacha Marlov)" `
     -KeyUsage DigitalSignature `
     -KeyAlgorithm RSA `
     -KeyLength 4096 `
     -NotAfter (Get-Date).AddYears(1) `
     -CertStoreLocation "Cert:\CurrentUser\My"
   ```

   The cert thumbprint goes to a 1Password vault item, never to
   git. The exported `.pfx` file (with passphrase) is uploaded as
   the GitHub Actions secret `WINDOWS_CERT_PFX_BASE64` +
   `WINDOWS_CERT_PASSWORD`.

2. **CI signing step** in `.github/workflows/release.yml`
   (Windows runner only):

   ```yaml
   - name: Sign Windows binary
     if: env.WINDOWS_CERT_PFX_BASE64 != ''
     shell: pwsh
     run: |
       $pfx = [Convert]::FromBase64String($env:WINDOWS_CERT_PFX_BASE64)
       Set-Content -Path cert.pfx -Value $pfx -AsByteStream
       signtool sign /f cert.pfx /p $env:WINDOWS_CERT_PASSWORD `
         /tr http://timestamp.digicert.com /td sha256 /fd sha256 `
         "src-tauri/target/release/bundle/nsis/*.exe"
       Remove-Item cert.pfx
   ```

   The step is gated on the secret being set. When unset (forks,
   PRs from contributors), the build still produces an unsigned
   `.exe` rather than failing — Authenticode is a release-line
   concern, not a green-CI gate.

3. **Verification** — every release artifact has a
   `signtool verify /pa /v` step that asserts the signature is
   present and chains to the local Trusted Publishers store.
   This catches "we forgot to sign" regressions.

4. **Demo machine prep** — the README's "Run the demo" section
   tells the operator to import `dev-cert.crt` (the public cert,
   exported from the cert store, no private key) into the local
   `Trusted Publishers` store before the first launch. The
   command is `Import-Certificate -FilePath dev-cert.crt
-CertStoreLocation Cert:\CurrentUser\TrustedPublisher`.

### Explicit triggers for upgrading to OV/EV

- A first non-academic user reports the SmartScreen warning as a
  blocker on a fresh Windows install.
- The project transitions from academic deliverable to public
  release (any tag ≥ `v1.0.0`).
- Microsoft tightens SmartScreen on unsigned/self-signed binaries
  (e.g., Windows 12 default-deny policy).

## Consequences

**Positive**

- The Phase I deliverable carries a real Authenticode signature.
  Reviewer can point `signtool verify /pa /v` at it on the demo
  machine and see a green check.
- Zero-cost, reproducible: the cert and the signing pipeline can
  be regenerated from documented commands.
- Honest about its limits — no claim of public-release-grade
  signing, no false reputation accrual.

**Negative**

- Other Windows machines (not the demo) trigger SmartScreen.
  Documented in README so this is no surprise.
- One-year cert expiry creates a future maintenance ticket.
- The signing step can become a CI flaky point if the timestamp
  server (`timestamp.digicert.com`) hiccups. Mitigated by retry
  logic in the GitHub Actions step.
